# 系统维护检查清单

> 当新一轮开发状态为「进行中」时，定时任务执行此清单
> 维护目标：保持系统健康、检测漂移、自动修复、采集指标

> **云端存储**：本清单里提到的所有文件（`进度表.md`、项目源码、维护记录）都从 GitHub 仓库 `MrZ-zhy/HJB` 的 `trae/solo-agent-TbCBsF` 分支同步而来，clone 后的工作区路径是 `/workspace/HJB/核聚变开源贡献系统/`。所有维护结果必须通过 `git add` + `git commit` + `git push` 写回 GitHub 才算维护完成。

## 检查流程

### 第一步：进度表完整性验证

- [ ] 进度表.md文件存在且可读（`/workspace/HJB/核聚变开源贡献系统/进度表.md`）
- [ ] 新一轮开发状态字段存在且值为「进行中」或「已完成」
- [ ] 当前开发项目信息完整（项目名称、仓库、分支）
- [ ] 链状开发进度树结构完整
- [ ] 操作指令区字段完整（NEXT_ACTION, CURRENT_CHAIN_NODE, DEVELOP_PHASE, LOCK）
- [ ] 检查点字段存在且指向有效检查点（LAST_CHECKPOINT）
- [ ] WIP状态字段存在（WIP_STATUS）

### 第二步：项目源码同步

- [ ] 项目目录存在（`/workspace/HJB/项目/<name>/`）
- [ ] 执行 `git fetch origin` 获取远程更新
- [ ] 检查是否有新commit：`git log HEAD..origin/main --oneline`
- [ ] 如有新commit，执行 `git pull --rebase`
- [ ] 检查是否有冲突，如有则记录在链条中
- [ ] 检查特性分支是否落后upstream/main超过10个commit

### 第三步：分支状态检查

- [ ] 当前分支名与进度表记录一致
- [ ] 分支未落后远程分支
- [ ] 工作区干净（无未提交修改）
- [ ] 特性分支存活时间未超过48小时（超时触发rebase或重建）

### 第四步：PR状态跟踪与漂移检测

> 漂移检测：对比期望状态与实际状态，发现不一致时触发收敛。

- [ ] 检查已提交PR的状态（merged/closed/open）
- [ ] 检查PR是否有新的Review Comment
- [ ] 检查CI状态（passing/failing/pending）
- [ ] 检查PR是否可合并（无冲突）
- [ ] 对比期望状态与实际状态，记录漂移事件

#### 漂移检测矩阵

| 期望状态 | 实际状态 | 漂移类型 | 自动收敛操作 |
|---------|---------|---------|-------------|
| PR open, CI pending | PR open, CI failing | CI失败 | 分析日志，可自动修复则修复并推送 |
| PR open, 可合并 | PR open, 有冲突 | 合并冲突 | 自动rebase解决冲突 |
| PR open | PR closed | PR被关闭 | 分析关闭原因，决定重新提交或放弃 |
| PR open, 无新评论 | PR open, 有新Review | 新Review | 记录反馈，等待开发轮次响应 |
| 分支与main同步 | 分支落后main | 分支漂移 | 自动rebase到最新main |
| Issue open | Issue closed | Issue已关闭 | 评估是否仍需贡献，可能需要切换Issue |

### 第五步：协调循环执行

> 参考Kubernetes协调循环：检测到漂移后，自动执行收敛操作使实际状态趋向期望状态。

- [ ] 遍历所有漂移事件
- [ ] 对每个漂移事件执行对应的自动收敛操作
- [ ] 记录收敛操作结果（成功/失败/需人工介入）
- [ ] 收敛失败的漂移事件升级为告警

#### 协调循环安全边界

- 单次维护周期内，同一PR最多执行2次收敛操作
- force-push操作需记录原始commit SHA
- 收敛操作不得修改PR的代码逻辑（仅限rebase/lint修复）
- 连续3次收敛失败后，锁定该PR并告警

### 第六步：自愈触发检查

> 参考Kubernetes自愈模式：对特定异常自动触发修复。

- [ ] 检查是否有CI lint/style失败 → 自动修复格式问题
- [ ] 检查是否有分支漂移 → 自动rebase
- [ ] 检查是否有进度表与实际不一致 → 以实际状态为准更新
- [ ] 检查自愈操作是否连续失败3次 → 如是，停止自愈并告警

#### 自愈操作记录

每次自愈操作需记录：

```yaml
自愈ID: heal-20260614-001
触发时间: 2026-06-14 10:30
触发条件: CI lint失败
操作: 自动修复格式问题并force-push
结果: 成功/失败
原始commit: abc1234
修复后commit: def5678
```

### 第七步：贡献指标采集

> 参考RED方法（Rate/Errors/Duration）：采集关键指标，用于SLO监控和策略优化。

- [ ] 采集PR提交率（Rate）：本周新提交PR数
- [ ] 采集错误率（Errors）：CI失败率、PR拒绝率
- [ ] 采集延迟（Duration）：从Issue分析到PR合并的平均时长
- [ ] 更新贡献指标到进度表

#### 指标采集项

| 指标 | 计算方式 | 采集频率 | 用途 |
|------|---------|---------|------|
| PR合并率 | 已合并PR数 / 总提交PR数 | 每次维护 | SLO监控、策略调整 |
| CI首次通过率 | CI首次通过PR数 / 总提交PR数 | 每次维护 | 代码质量评估 |
| 平均合并时长 | PR从创建到合并的平均时间 | 每次维护 | 流程效率评估 |
| Review响应时间 | 维护者首次Review的平均时间 | 每次维护 | 仓库活跃度评估 |
| 代码退回率 | 要求重大修改的PR数 / 总提交PR数 | 每次维护 | 代码质量评估 |
| 自愈成功率 | 自愈成功次数 / 自愈触发次数 | 每次维护 | 自愈机制健康度 |

### 第八步：SLO监控与错误预算

> 参考SRE错误预算：监控SLO合规性，预算耗尽时触发可靠性模式。

- [ ] 计算当前错误预算消耗情况
- [ ] 检查是否有SLO被违反
- [ ] 如错误预算耗尽，标记系统进入可靠性模式

#### SLO燃烧率告警

| SLO | 燃烧率阈值 | 告警级别 | 处理 |
|-----|-----------|---------|------|
| PR合并率≥30% | 1小时内消耗>10%预算 | P1 | 暂停新PR提交，分析原因 |
| CI首次通过率≥80% | 1小时内消耗>10%预算 | P1 | 暂停新PR提交，检查CI环境 |
| Review响应≤48h | 连续3个PR超时 | P2 | 降低该仓库优先级 |
| 代码退回率≤10% | 连续2个PR被退回 | P2 | 切换到保守策略 |

### 第九步：更新维护状态

- [ ] 更新进度表中的系统维护状态表
- [ ] 更新 LAST_UPDATE 日期
- [ ] 更新贡献指标数据
- [ ] 更新错误预算状态
- [ ] 如发现异常，在操作指令区标记
- [ ] 记录本次维护的漂移事件数和收敛操作数

## 异常处理

| 异常情况 | 处理方式 | 自愈 | 升级条件 |
|---------|---------|------|---------|
| 进度表损坏 | 从备份/检查点恢复或重建 | 是 | 恢复失败→锁定系统 |
| 源码同步冲突 | 记录冲突，等待开发轮次处理 | 否 | 冲突持续>48h→告警 |
| PR被关闭 | 分析原因，决定是否重新提交 | 否 | 连续2个PR被关闭→切换仓库 |
| 分支丢失 | 从远程重新拉取 | 是 | 远程也丢失→从检查点恢复 |
| 定时任务失败 | 检查LOCK状态，必要时重置 | 是 | 连续3次失败→锁定系统 |
| CI持续失败 | 分析日志，自动修复lint问题 | 部分 | 逻辑错误→回退到开发轮次 |
| 漂移无法收敛 | 记录漂移，等待人工介入 | 否 | 漂移持续>24h→锁定系统 |
| 错误预算耗尽 | 进入可靠性模式，暂停新PR | 否 | 需人工分析并重置预算 |
| 自愈连续失败 | 停止自愈，锁定相关PR | 否 | 需人工介入修复 |

## 告警分级

| 级别 | 定义 | 响应时限 | 通知方式 |
|------|------|---------|---------|
| P0 | 系统锁定，无法自动恢复 | 立即 | 进度表标记+人工介入 |
| P1 | SLO严重违反，错误预算快速消耗 | 24小时内 | 进度表标记 |
| P2 | 单个PR/仓库异常，不影响全局 | 下次维护时 | 记录在案 |
| P3 | 信息性事件，无需立即处理 | 无 | 记录在案 |

## 维护完成标志

所有检查项完成后：
1. 在进度表系统维护状态中更新时间戳
2. 记录本次维护摘要（漂移数、收敛数、自愈数、指标快照）
3. 不修改新一轮开发状态（保持「进行中」）
4. 如有P0/P1告警，在操作指令区标记
5. **强制要求：执行 `git add . && git commit && git push` 写回 GitHub**
