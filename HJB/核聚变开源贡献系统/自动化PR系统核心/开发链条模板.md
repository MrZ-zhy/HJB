# 开发链条模板

> 此模板定义了每个项目的标准开发链条结构。
> 实际开发时根据项目特点调整，但必须遵循此框架。
>
> **本文件存储在云端**：`MrZ-zhy/HJB` · `trae/solo-agent-TbCBsF` 分支，路径 `/workspace/HJB/核聚变开源贡献系统/自动化PR系统核心/开发链条模板.md`。链条对应的代码工作目录是 `/workspace/HJB/项目/<name>/`（也是从 GitHub clone 下来的）。每个节点产出 → 即改即 commit；整轮结束 → push 回云端。

## 标准开发链条

```
[项目根]
 │
 ├──⏳ P1-环境搭建 ── fork仓库、clone源码、配置开发环境
 │   │
 │   ├──⏳ P1.1-代码研读 ── 阅读项目结构、核心模块、文档
 │   │   │
 │   │   ├──⏳ P1.1.1-架构理解 ── 绘制模块依赖关系图
 │   │   │   │
 │   │   │   └──⏳ P1.1.2-Issue筛选 ── 找到适合AI贡献的Issue
 │   │   │
 │   │   └──⏳ P1.2-测试验证 ── 在 clone 的工作目录运行上游测试套件，确认环境可用
 │   │       📌 检查点：环境就绪确认
 │   │
 │   ├──⏳ P2-开发实施 ── 根据选定Issue进行开发
 │   │   │
 │   │   ├──⏳ P2.1-方案设计 ── 设计实现方案，记录在链条中
 │   │   │   │
 │   │   │   ├──⏳ P2.1.1-主方案 ── 首选实现路径
 │   │   │   │   └──🔀 P2.1.2-备选方案 ── 如主方案受阻的替代路径
 │   │   │   │
 │   │   │   └──⏳ P2.2-编码实现 ── 编写代码
 │   │   │       🚧 质量门禁：lint通过、无硬编码、类型注解完整
 │   │   │
 │   │   ├──⏳ P2.3-测试验证 ── 运行测试、手动验证
 │   │   │   🚧 质量门禁：新增测试覆盖变更、全部测试通过
 │   │   │   📌 检查点：测试通过确认
 │   │   │
 │   │   └──⏳ P2.4-代码审查（反思循环）── Critic Agent自审代码质量
 │   │       🚧 质量门禁：0 critical issue
 │   │       🔄 反思循环：不通过则自动修订（最多3轮）
 │   │       📌 检查点：审查通过确认
 │   │
 │   ├──⏳ P3-PR提交 ── 创建Pull Request
 │   │   │
 │   │   ├──⏳ P3.1-分支整理 ── 整理commit、rebase
 │   │   │   🚧 质量门禁：Conventional Commits格式、无merge commit
 │   │   │
 │   │   └──⏳ P3.2-PR创建 ── 推送分支、创建PR
 │   │       🚧 质量门禁：PR描述完整、关联Issue、测试方法明确
 │   │       📌 检查点：PR创建成功
 │   │
 │   └──⏳ P4-后续跟进 ── 跟踪PR审核状态
 │       │
 │       ├──⏳ P4.1-响应反馈 ── 根据review意见修改
 │       │   🔄 反思循环：修改后再次Critic审查
 │       │
 │       └──⏳ P4.2-合并确认 ── 确认PR被合并
 │           🧹 合并后清理：删除特性分支
```

## 链条生长规则

1. **顺序推进**：默认按P1→P2→P3→P4顺序推进
2. **条件分支**：当主方案受阻时，创建🔀分支节点
3. **分支合并**：分支成功后合并回主线，失败则标记❌
4. **回溯**：允许回退到上一节点重新执行
5. **并行**：同一层级的不同分支可并行（但AI一次只执行一条路径）
6. **质量门禁阻断**：未通过门禁的节点不得标记为✅，必须修复或回退
7. **检查点保护**：回退时优先回到最近的📌检查点，而非从头开始
8. **反思循环**：P2.4和P4.1节点内置反思循环，最多3轮自动修订

## 反思循环详细流程

> 参考AI Agent框架的Draft→Critique→Revise模式

```
┌─────────────────────────────────────────────┐
│            反思循环（最多3轮）                 │
│                                             │
│  ┌──────────┐    ┌──────────┐    ┌────────┐ │
│  │ Draft    │───→│ Critic   │───→│ Revise │ │
│  │ 当前代码  │    │ 审查代码  │    │ 修订   │ │
│  └──────────┘    └────┬─────┘    └───┬────┘ │
│                       │               │      │
│                  ┌────▼────┐          │      │
│                  │ 0 crit? │──Yes──→ ✅通过  │
│                  └────┬────┘          │      │
│                       │ No            │      │
│                       ▼               │      │
│                  轮次<3? ──Yes────────┘      │
│                       │ No                   │
│                       ▼                      │
│                  ❌失败，回退到P2.1            │
└─────────────────────────────────────────────┘
```

### Critic Agent审查清单

| 维度 | 检查项 | 严重级别 |
|------|--------|---------|
| 正确性 | 逻辑是否正确，边界条件是否处理 | Critical |
| 正确性 | 是否有空指针/索引越界风险 | Critical |
| 风格 | 是否与项目现有代码风格一致 | Warning |
| 风格 | 命名是否清晰、符合项目惯例 | Warning |
| 测试 | 变更代码是否有对应测试 | Critical |
| 测试 | 测试是否覆盖正常和异常路径 | Warning |
| 安全 | 是否引入安全风险 | Critical |
| 性能 | 是否引入明显性能退化 | Warning |
| 核工程 | 物理单位是否一致 | Critical |
| 核工程 | 数值精度是否足够 | Critical |
| 核工程 | 安全裕度是否保守 | Warning |

### 修订规则

- Critical级别问题必须修复，否则不得通过
- Warning级别问题尽量修复，3轮后仍有Warning可放行（记录在案）
- 每轮修订后重新运行测试，确保未引入新问题
- 修订不得改变方案设计的核心思路（如需改变，应回退到P2.1）

## 回退与恢复规则

| 场景 | 回退目标 | 恢复操作 |
|------|---------|---------|
| P2.2编码失败（3轮反思未通过） | P2.1方案设计 | 重新设计方案，可能切换到备选方案 |
| P2.3测试失败（无法修复） | P2.2编码实现 | 修改代码实现 |
| P2.4审查失败（3轮反思未通过） | P2.1方案设计 | 方案本身有问题，需重新设计 |
| P3.1分支整理失败 | P2.4审查 | 重新审查提交历史 |
| P4.1 Review要求重大修改 | P2.1方案设计 | 重新设计方案 |
| P4.1 Review要求小修改 | P2.2编码实现 | 直接修改代码 |
| 系统异常中断 | 最近📌检查点 | 从检查点恢复状态 |

## 链条与进度表的映射

- 链条当前活跃节点 → 进度表的 `CURRENT_CHAIN_NODE`
- 链条整体阶段 → 进度表的 `DEVELOP_PHASE`
- 链条完成状态 → 进度表的 `新一轮开发状态`
- 链条最新检查点 → 进度表的 `LAST_CHECKPOINT`（新增字段）
- 链条WIP状态 → 进度表操作指令区 `WIP_STATUS`（新增字段）
