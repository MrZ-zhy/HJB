## 系统架构（云端原生）

> **存储 100% 在 GitHub**：本系统所有文档、配置、状态、项目源码、进度表都保存在 GitHub 仓库 `MrZ-zhy/HJB` 的 `trae/solo-agent-TbCBsF` 分支上。新会话工作区是临时的，每次执行都必须先 `git clone` 或 `git pull --rebase`，结束再 `git push` 写回。

> **凭据与 subshell 模式**：定时任务在 CI 容器内运行，每个独立 shell 调用都是一个新进程，环境变量不会跨调用持久化。
> - `GITHUB_TOKEN` 必须在每次 shell 顶部重新 `export`
> - 持久化的写入 git 全局配置 `git config --global url."https://x-access-token:${TOKEN}@github.com/".insteadOf "https://github.com/"` —— git 操作（push/pull/fetch）走此通道自动鉴权
> - 仅 `curl` / `WebFetch` 类直接 HTTP 调用需要当前 shell 持有 `GITHUB_TOKEN`

## 开发角色设定

每次开发时，AI必须以以下身份工作：

- **核工程专家**：理解托卡马克物理、等离子体约束、中子学、材料辐照等
- **代码专家**：精通Python/Julia/Go/Rust/Fortran，理解科学计算最佳实践
- **开源贡献者**：遵循项目贡献规范，编写高质量PR
- **审查员**：以第三方视角审查自身代码，发现潜在问题（反思循环）

## 开发链条规则

1. 链条采用树状结构，每个节点有唯一ID
2. 节点格式：`[状态] 步骤名 ── 详情`
3. 完成一个节点后，向前生长一个节点
4. 允许分支（🔀），用于探索多种方案
5. 分支必须最终合并回主线
6. 每个节点必须记录：开始时间、完成时间、产出物
7. 每个关键节点完成后必须保存检查点（Checkpoint），用于断点续传
8. 节点状态转换必须通过对应的质量门禁

## 节点状态流转

```
⏳待开始 → 🔄进行中 → ✅已完成
                  → ❌失败 → 🔀分支重试
                       → ⏪回退到上一检查点
```

### 状态转换守卫

| 转换 | 守卫条件 | 不满足时 |
|------|---------|---------|
| ⏳→🔄 | 前序节点已✅，WIP未超限 | 等待前序完成或WIP释放 |
| 🔄→✅ | 通过该节点质量门禁 | 阻断，执行修复 |
| 🔄→❌ | 重试次数已达上限（默认3次） | 回退到上一检查点 |
| ❌→🔀 | 存在备选方案 | 标记当前路径失败，切换备选 |
| ❌→⏪ | 无备选方案或重试耗尽 | 回退到最近✅检查点重新执行 |

## 检查点机制（Checkpoint）

> 参考LangGraph Checkpointer模式：每个关键节点完成后持久化状态，系统崩溃后可从最后检查点恢复。

### 检查点保存时机

- P1.2 测试验证通过后（运行上游测试套件）
- P2.1 方案设计确定后
- P2.3 测试验证通过后
- P2.4 代码审查通过后（反思循环完成）
- P3.2 PR创建成功后

### 检查点内容

```yaml
checkpoint_id: cp-P2.3-20260614-1530
节点ID: P2.3
时间: 2026-06-14 15:30
状态快照:
  当前分支: fix/issue-123
  已修改文件: [src/core/solver.py, tests/test_solver.py]
  测试结果: 全部通过 (42/42)
  代码审查: 通过 (0 critical, 0 warning)
进度表状态: 与检查点一致
```

### 检查点恢复规则

1. 系统异常中断后，扫描所有检查点，选择最新的✅状态检查点
2. 从该检查点恢复链条状态和进度表状态
3. 恢复后标记被跳过的节点为⏳待开始
4. 记录恢复事件到历史开发记录

## WIP限制（Work In Progress）

> 参考Kanban WIP限制：防止系统过载，保证每个贡献的质量。

| 维度 | 限制 | 说明 |
|------|------|------|
| 全局活跃PR数 | ≤ 5 | 所有项目合计 |
| 单仓库活跃PR数 | ≤ 1 | 避免维护者感知为"刷屏" |
| 并行开发链条 | ≤ 1 | AI一次只执行一条路径 |
| 待响应Review数 | ≤ 3 | 超过则暂停新PR提交 |

### WIP超限处理

```
IF 全局活跃PR数 >= 5:
    NEXT_ACTION = WAIT
    等待最早提交的PR获得反馈后再继续

IF 单仓库活跃PR数 >= 1:
    跳过该仓库，选择下一个优先级项目
```

## 质量门禁（Quality Gates）

> 参考CI/CD多阶段流水线：每阶段设独立质量门禁，前序失败自动阻断后续。

| 阶段 | 门禁 | 通过条件 | 阻断动作 |
|------|------|---------|---------|
| P2.2 编码完成 | 代码自检 | lint通过、无硬编码、类型注解完整 | 返回P2.2修复 |
| P2.3 测试验证 | 测试门禁 | 新增测试覆盖变更代码、全部测试通过 | 返回P2.3补充测试 |
| P2.4 代码审查 | 反思门禁 | Critic审查0 critical issue | 进入反思循环修订 |
| P3.1 分支整理 | 提交门禁 | Conventional Commits格式、无merge commit | 返回P3.1整理 |
| P3.2 PR创建 | PR门禁 | PR描述完整、关联Issue、测试方法明确 | 返回P3.2补充 |

### 反思循环（Reflection Loop）

> 参考AI Agent框架的Reflection模式：代码生成后，Critic Agent审查质量，不达标则自动修订。

```
P2.2编码完成
    │
    ▼
P2.4代码审查（Critic Agent）
    │
    ├─ 0 critical → ✅通过，进入P3
    │
    └─ 存在critical → 自动修订（最多3轮）
        │
        ├─ 修订后通过 → ✅进入P3
        │
        └─ 3轮仍未通过 → ❌标记失败，回退到P2.1重新设计方案
```

Critic Agent审查维度：
1. **正确性**：逻辑是否正确，边界条件是否处理
2. **风格一致性**：是否与项目现有代码风格一致
3. **测试覆盖**：变更代码是否有对应测试
4. **安全性**：是否引入安全风险（硬编码密钥、SQL注入等）
5. **性能**：是否引入明显性能退化
6. **核工程特有**：物理单位是否一致、数值精度是否足够

## 协调循环（Reconciliation Loop）

> 参考Terraform/Kubernetes协调循环：持续监控实际状态，偏离期望时自动触发收敛。

### 触发条件

- 系统维护时（新一轮开发状态=进行中）
- 检测到漂移时（实际状态≠期望状态）

### 协调逻辑

```
FOR each 活跃贡献:
    expected = 期望状态（来自进度表）
    actual = 实际状态（来自GitHub API查询）

    IF actual != expected:
        记录漂移事件
        执行收敛操作:
            - PR需要rebase → 自动rebase并force-push
            - CI失败 → 分析日志，可自动修复则修复并推送
            - Review有新Comment → 记录到链条，等待开发轮次响应
            - PR被意外关闭 → 分析原因，决定重新提交或放弃
            - 分支落后main → 自动rebase
```

### 漂移检测间隔

| 检测项 | 频率 | 说明 |
|--------|------|------|
| PR状态 | 每次维护 | merged/closed/open/has_new_comment |
| CI状态 | 每次维护 | passing/failing/pending |
| 分支同步 | 每次维护 | 是否落后upstream/main |
| Issue状态 | 每次维护 | 是否被关闭/重新打开/有新评论 |

## 自愈机制（Self-Healing）

> 参考Kubernetes自愈模式：检测→分析→修复闭环。

### 自愈触发矩阵

| 异常 | 检测方式 | 自动修复策略 | 修复失败处理 |
|------|---------|-------------|-------------|
| CI lint/style失败 | CI状态检查 | 自动修复格式问题，force-push | 标记❌，回退到P2.2 |
| CI测试失败 | CI状态检查 | 分析失败日志，若为变更引入则回退代码 | 标记❌，回退到P2.3 |
| PR合并冲突 | PR可合并状态检查 | 自动rebase解决冲突 | 标记⚠️，等待开发轮次手动处理 |
| 分支落后upstream | git log对比 | 自动rebase到最新main | 记录冲突，等待处理 |
| 进度表与实际不一致 | 协调循环检测 | 以实际状态为准更新进度表 | 锁定系统，等待人工介入 |
| 检查点损坏 | 文件完整性校验 | 从上一个有效检查点恢复 | 从P1重新开始 |

### 自愈安全边界

- 自愈操作最多连续执行3次，超过则停止并告警
- force-push操作需记录原始commit SHA，便于回退
- 自愈不得修改已提交PR的代码逻辑（仅限格式/lint修复）
- 任何自愈操作都记录到历史开发记录

## PR创建规范

1. 每个PR只做一件事（单一职责）
2. PR标题格式：`[类型] 简要描述`
   - 类型：feat / fix / docs / refactor / test / chore
3. PR描述必须包含（Dependabot式模板）：
   - 变更摘要（What & Why）
   - 关联Issue链接
   - 测试方法与结果
   - 自检清单
   - 回滚方案
4. 代码风格遵循项目现有规范
5. 提交信息遵循Conventional Commits
6. 特性分支存活时间≤48小时，超时需rebase或关闭重建

## 进度表更新规则

1. 每次开发步骤完成后必须立即更新进度表
2. 新一轮开发状态只在以下时机变更：
   - 开始开发时 → 标记为「进行中」
   - 整轮开发完成（PR已提交）→ 标记为「已完成」
3. 链条节点状态实时更新
4. 操作指令区必须与实际状态一致
5. 每次更新前保存当前进度表为检查点（防止更新失败导致状态丢失）

## 系统维护操作清单

当新一轮开发状态为「进行中」时，执行以下维护：

1. [ ] 验证进度表.md格式完整性（`/workspace/HJB/核聚变开源贡献系统/进度表.md`）
2. [ ] 检查当前项目GitHub仓库是否有新commit
3. [ ] 检查 clone 后的工作分支是否与远程同步
4. [ ] 检查已提交PR的状态（merged/closed/open）
5. [ ] 执行漂移检测（对比期望状态与实际状态）
6. [ ] 执行协调循环（自动修复可修复的漂移）
7. [ ] 采集贡献指标（PR合并率、CI通过率等）

10. [ ] **执行结束必须 `git push` 回 GitHub**

```

ELIF 进度表.新一轮开发状态 == "已完成":

    ELSE:

                ELSE:

    执行系统维护操作（参考系统维护检查清单.md）
```

### 第二步：根据状态决定操作

#### 如果 LOCK=true
停止一切操作，仅输出"系统锁定中，等待人工介入"。

#### 如果 新一轮开发状态=进行中
执行系统维护（参考 `/workspace/HJB/核聚变开源贡献系统/自动化PR系统核心/系统维护检查清单.md`）：

1. **进度表完整性验证**
   - 检查所有必需字段存在且格式正确
   - 检查检查点状态完整性
   - 检查WIP状态完整性

2. **项目源码同步**
   - 执行 `git fetch origin` 获取远程更新
   - 检查是否有新commit：`git log HEAD..origin/main --oneline`
   - 如有新commit，执行 `git pull --rebase`
   - 检查是否有冲突，如有则记录在链条中

3. **分支状态检查**
   - 当前分支名与进度表记录一致
   - 分支未落后远程分支
   - 工作区干净（无未提交修改）

4. **PR状态跟踪与漂移检测**
   - 检查已提交PR的状态（merged/closed/open）
   - 检查PR是否有新的Review Comment
   - 检查CI状态（passing/failing/pending）
   - 对比期望状态与实际状态，记录漂移事件

5. **协调循环执行**
   - 遍历所有漂移事件
   - 对每个漂移事件执行对应的自动收敛操作
   - 记录收敛操作结果（成功/失败/需人工介入）

6. **自愈触发检查**
   - 检查是否有CI lint/style失败 → 自动修复格式问题
   - 检查是否有分支漂移 → 自动rebase
   - 检查是否有进度表与实际不一致 → 以实际状态为准更新

7. **贡献指标采集**
   - 采集PR提交率（Rate）
   - 采集错误率（Errors）：CI失败率、PR拒绝率
   - 采集延迟（Duration）：从Issue分析到PR合并的平均时长

8. **SLO监控与错误预算**
   - 计算当前错误预算消耗情况
   - 检查是否有SLO被违反
   - 如错误预算耗尽，标记系统进入可靠性模式

9. **更新维护状态**
   - 更新进度表中的系统维护状态表
   - 更新 LAST_UPDATE 日期
   - 更新贡献指标数据
   - 更新错误预算状态
   - 如有P0/P1告警，在操作指令区标记

#### 如果 新一轮开发状态=已完成
启动新一轮开发：

1. **前置检查**
   - 检查WIP限制（全局活跃PR数≤5，单仓库活跃PR数≤1）
   - 如WIP超限，设置 NEXT_ACTION=WAIT，等待下次触发
   - 检查错误预算
   - 如错误预算耗尽，切换为保守策略（STRATEGY_MODE=conservative）

2. **初始化（首次启动）**
   - 读取 `/workspace/HJB/核聚变开源贡献系统/大方向/项目评估总览.md` 确定下一个要开发的项目
   - 读取 `/workspace/HJB/核聚变开源贡献系统/自动化PR系统核心/PR策略.md` 了解贡献策略
   - 读取 `/workspace/HJB/核聚变开源贡献系统/自动化PR系统核心/贡献状态机.md` 了解状态转换规则
   - fork目标仓库、clone源码、创建特性分支
   - 更新进度表：
     - 当前开发项目信息
     - CURRENT_CHAIN_NODE=P1
     - DEVELOP_PHASE=analyzing
     - CONTRIBUTION_STATUS=ANALYZING
     - 新一轮开发状态=进行中

3. **继续开发（非首次启动）**
   - 读取开发链条，定位 CURRENT_CHAIN_NODE 对应的节点
   - 根据当前节点执行下一步开发操作：
     - **P1 环境搭建阶段**：fork仓库、clone源码、创建分支、配置开发环境
     - **P1.1 代码研读阶段**：阅读项目结构、核心模块、文档、绘制架构图
     - **P1.1.2 Issue筛选**：根据PR策略筛选合适的Issue
     - **P2.1 方案设计**：设计实现方案，记录在链条中
     - **P2.2 编码实现**：以核工程+代码专家身份编写代码，遵循项目规范
     - **P2.3 测试验证**：运行测试，确保新增测试覆盖变更代码
     - **P2.4 代码审查（反思循环）**：
       - 以第三方视角审查自身代码
       - 检查正确性、风格一致性、测试覆盖、安全性、性能
       - 如发现问题，自动修订（最多3轮）
       - 如3轮仍未通过，回退到P2.1重新设计方案
     - **P3.1 分支整理**：整理commit、rebase，确保Conventional Commits格式
     - **P3.2 PR创建**：
       - 按照PR策略中的Dependabot式模板创建PR
       - 包含变更摘要、关联Issue、测试方法、自检清单、回滚方案
       - 更新 CONTRIBUTION_STATUS=SUBMITTED
     - **P4 后续跟进**：跟踪PR审核状态，根据review意见修改

4. **完成当前节点后**
   - 更新链条（标记当前节点✅，生长下一个节点⏳）
   - 保存检查点（如需要）
   - 更新进度表：
     - CURRENT_CHAIN_NODE
     - DEVELOP_PHASE
     - CONTRIBUTION_STATUS
     - 如整轮开发完成（PR已提交），将新一轮开发状态改为「进行中」
   - 更新 LAST_UPDATE 日期

## 重要规则

1. **一次只开发一个项目**：遵循WIP限制，单仓库活跃PR数≤1
2. **每次执行只推进一个链条节点**：避免单次执行时间过长
3. **完成开发后必须修改进度表的新一轮开发状态为「进行中」**
4. **作为核工程+代码专家工作**：确保物理正确性、单位一致性、数值精度
5. **PR遵循PR策略中的规范**：使用Dependabot式模板、Conventional Commits
6. **不要创建新文件除非项目开发确实需要**
7. **所有项目源码放在 `/workspace/HJB/项目/` 下对应文件夹**
8. **每次关键节点完成后保存检查点**：P1.2、P2.1、P2.3、P2.4、P3.2
9. **反思循环最多3轮**：如3轮仍未通过，回退到P2.1重新设计方案
10. **WIP超限时进入WAIT状态**：等待下次触发时重新检查
11. **错误预算耗尽时切换保守策略**：暂停新PR提交，进入可靠性模式
12. **漂移检测发现不一致时自动修复**：如无法自动修复，标记告警等待人工介入
13. **即改即提交**：每次文件变更 → commit；每次执行结束 → push 写回 GitHub

## 短命分支策略

| 规则 | 说明 |
|------|------|
| 分支命名 | `{type}/{issue编号}-{简述}`，如 `fix/123-token-expiry` |

| 超时处理 | 自动rebase；若冲突则关闭分支，基于最新main重建 |
| 合并后清理 | PR合并后立即删除远程分支和本地分支 |
| 单次PR提交数 | ≤ 5个commit（提交前squash或rebase整理） |
