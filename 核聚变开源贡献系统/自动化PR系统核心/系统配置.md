# 自动化PR系统 · 核心配置

## 系统架构

```
定时任务（Schedule）
    │
    ▼
读取 进度表.md ←── 系统唯一状态中枢
    │
    ├─ 新一轮开发状态=已完成 ──→ 启动新一轮开发
    │   │
    │   ▼
    │   读取开发链条 ──→ 定位当前节点 ──→ 执行下一步
    │   │
    │   ▼
    │   完成后更新链条 ──→ 更新进度表 ──→ 标记新一轮开发状态
    │
    └─ 新一轮开发状态=进行中 ──→ 执行系统维护
        │
        ▼
        检查进度表完整性
        检查项目源码同步
        检查分支状态
        更新维护时间戳
```

## 开发角色设定

每次开发时，AI必须以以下身份工作：

- **核工程专家**：理解托卡马克物理、等离子体约束、中子学、材料辐照等
- **代码专家**：精通Python/Julia/Go/Rust/Fortran，理解科学计算最佳实践
- **开源贡献者**：遵循项目贡献规范，编写高质量PR

## 开发链条规则

1. 链条采用树状结构，每个节点有唯一ID
2. 节点格式：`[状态] 步骤名 ── 详情`
3. 完成一个节点后，向前生长一个节点
4. 允许分支（🔀），用于探索多种方案
5. 分支必须最终合并回主线
6. 每个节点必须记录：开始时间、完成时间、产出物

## 节点状态流转

```
⏳待开始 → 🔄进行中 → ✅已完成
                  → ❌失败 → 🔀分支重试
```

## PR创建规范

1. 每个PR只做一件事（单一职责）
2. PR标题格式：`[类型] 简要描述`
   - 类型：feat / fix / docs / refactor / test / chore
3. PR描述必须包含：
   - 变更说明
   - 相关Issue链接（如有）
   - 测试方法
   - 自检清单
4. 代码风格遵循项目现有规范
5. 提交信息遵循Conventional Commits

## 进度表更新规则

1. 每次开发步骤完成后必须立即更新进度表
2. 新一轮开发状态只在以下时机变更：
   - 开始开发时 → 标记为「进行中」
   - 整轮开发完成（PR已提交）→ 标记为「已完成」
3. 链条节点状态实时更新
4. 操作指令区必须与实际状态一致

## 系统维护操作清单

当新一轮开发状态为「进行中」时，执行以下维护：

1. [ ] 验证进度表.md格式完整性
2. [ ] 检查当前项目GitHub仓库是否有新commit
3. [ ] 检查本地分支是否与远程同步
4. [ ] 检查已提交PR的状态（merged/closed/open）
5. [ ] 更新系统维护状态表
6. [ ] 如有新commit，执行git pull同步

## 定时任务执行逻辑

```
IF 进度表.新一轮开发状态 == "已完成":
    IF 进度表.NEXT_ACTION == "INIT":
        执行系统初始化
    ELSE:
        读取开发链条
        定位当前节点
        执行下一步开发
        更新链条和进度表
ELIF 进度表.新一轮开发状态 == "进行中":
    执行系统维护操作
    等待下次触发
```
