# 贡献状态机

> 参考LangGraph图状态机与Kanban工作流状态机：定义贡献任务的完整生命周期，约束合法状态转换，防止非法跳转。
>
> **本文件存储在云端**：`MrZ-zhy/HJB` · `trae/solo-agent-TbCBsF` 分支，路径 `/workspace/HJB/核聚变开源贡献系统/自动化PR系统核心/贡献状态机.md`。状态字段由 `进度表.md` 中的 `CONTRIBUTION_STATUS` 字段承载；状态变更 → 即改即 commit + push。

## 状态定义

| 状态 | 标记 | 说明 | 对应链条阶段 |
|------|------|------|-------------|
| 待分析 | 📋 BACKLOG | Issue已识别，尚未开始分析 | P1.1.2之前 |
| 分析中 | 🔍 ANALYZING | 正在分析Issue和代码库 | P1.1 |
| 待开发 | 📝 TODO | 分析完成，等待开发资源 | P2之前 |
| 开发中 | 💻 CODING | 正在编码实现 | P2.2 |
| 自审中 | 👁 REVIEWING | Critic Agent审查代码 | P2.4 |
| 待提交 | 📦 READY | 代码审查通过，等待提交PR | P3之前 |
| 等待中 | ⏸️ WAIT | WIP超限或错误预算耗尽，暂停提交 | 任意阶段 |
| 已提交 | 📮 SUBMITTED | PR已创建，等待CI | P3.2 |
| CI中 | ⚙️ CI_RUNNING | CI正在运行 | P3.2后 |
| Review中 | 💬 UNDER_REVIEW | 维护者正在Review | P4 |
| 待修订 | 🔧 REVISION | Review要求修改 | P4.1 |
| 已合并 | ✅ MERGED | PR已被合并 | P4.2 |
| 已拒绝 | ❌ REJECTED | PR被关闭/拒绝 | 终态 |
| 已取消 | 🚫 CANCELLED | 主动取消贡献 | 终态 |

## 合法状态转换表

| 从 | 到 | 触发条件 | 守卫条件 |
|----|-----|---------|---------|
| 📋 BACKLOG | 🔍 ANALYZING | 定时任务选择该Issue | WIP未超限 |
| 📋 BACKLOG | 📝 TODO | **主动贡献路径**：上游 0 open issues，定时任务自检发现代码缺陷（typo/低风险改进） | 变更范围明确、风险低 |
| 🔍 ANALYZING | 📝 TODO | 分析完成，确认Issue可贡献 | Issue仍open |
| 🔍 ANALYZING | 🚫 CANCELLED | Issue不合适 | 记录取消原因 |
| 📝 TODO | 💻 CODING | 开始编码 | 开发资源可用 |
| 💻 CODING | 👁 REVIEWING | 编码完成 | 通过代码自检门禁 |
| 💻 CODING | 📝 TODO | 编码失败3轮 | 记录失败原因 |
| 👁 REVIEWING | 📦 READY | Critic审查通过 | 0 critical issue |
| 👁 REVIEWING | 💻 CODING | Critic发现critical | 反思轮次<3 |
| 👁 REVIEWING | 📝 TODO | 反思3轮未通过 | 记录方案问题 |
| 📦 READY | 📮 SUBMITTED | 提交PR | WIP未超限 |
| 📦 READY | ⏸️ WAIT | WIP超限或错误预算耗尽 | 等待WIP释放或预算恢复 |
| ⏸️ WAIT | 📮 SUBMITTED | WIP释放或预算恢复 | WIP未超限且预算未耗尽 |
| 📦 READY | 📦 READY | WIP超限 | 等待WIP释放 |
| 📮 SUBMITTED | ⚙️ CI_RUNNING | PR创建成功 | — |
| ⚙️ CI_RUNNING | 💬 UNDER_REVIEW | CI通过 | — |
| ⚙️ CI_RUNNING | 💬 UNDER_REVIEW | **无 CI 配置**（仓库无 workflow） | 跳过 CI 阶段；视为人工 review |
| ⚙️ CI_RUNNING | 🔧 REVISION | CI失败(lint/style) | 可自动修复 |
| ⚙️ CI_RUNNING | 💻 CODING | CI失败(逻辑错误) | 需要代码修改 |
| 💬 UNDER_REVIEW | ✅ MERGED | PR被合并 | — |
| 💬 UNDER_REVIEW | 🔧 REVISION | Review要求修改 | — |
| 💬 UNDER_REVIEW | ❌ REJECTED | PR被关闭 | 记录拒绝原因 |
| 🔧 REVISION | 👁 REVIEWING | 修订完成 | 通过代码自检门禁 |
| 🔧 REVISION | ❌ REJECTED | 修订失败 | 记录失败原因 |

## 非法状态转换

以下转换被禁止，如检测到则表明系统状态异常：

| 非法转换 | 原因 |
|---------|------|
| 📋 BACKLOG → 💻 CODING | 必须先分析Issue |
| 💻 CODING → 📮 SUBMITTED | 必须先通过Critic审查 |
| 👁 REVIEWING → 📮 SUBMITTED | 必须先通过审查门禁 |
| ⚙️ CI_RUNNING → ✅ MERGED | CI通过后还需Review |
| 🔧 REVISION → 📮 SUBMITTED | 修订后必须重新审查 |
| ✅ MERGED → 任何状态 | 终态不可逆转 |
| ❌ REJECTED → 任何状态 | 终态不可逆转 |

## 多贡献并发状态管理

当系统同时跟踪多个贡献时（如1个PR在Review中，1个在开发中）：

```
贡献A: 💬 UNDER_REVIEW → 系统维护模式跟踪
贡献B: 💻 CODING → 开发模式执行

系统状态 = MAX(所有贡献的状态优先级)

状态优先级（从高到低）：
🔧 REVISION > 💻 CODING > 👁 REVIEWING > ⚙️ CI_RUNNING
> 💬 UNDER_REVIEW > 📮 SUBMITTED > 📦 READY > 📝 TODO > 📋 BACKLOG

系统根据最高优先级贡献的状态决定 NEXT_ACTION
```

## 状态持久化

每次状态转换后，必须同步更新：

1. **进度表.md** - 更新操作指令区
2. **开发链条** - 更新对应节点状态
3. **检查点** - 关键状态转换时保存检查点
4. **历史记录** - 终态（MERGED/REJECTED/CANCELLED）时记录到历史开发记录

### 持久化顺序

```
1. 保存检查点（如需要）
2. 更新开发链条节点状态
3. 更新进度表操作指令区
4. 验证三者一致性
5. 如不一致，以开发链条为准进行修正
```
