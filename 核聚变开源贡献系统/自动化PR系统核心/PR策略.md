# PR策略

## 贡献类型优先级

> 按AI可完成度和贡献价值排序

| 优先级 | 贡献类型 | 说明 | AI适合度 |
|--------|---------|------|---------|
| 1 | Bug修复 | 修复已知bug，范围明确 | ★★★★★ |
| 2 | 文档完善 | 补充API文档、教程、示例 | ★★★★★ |
| 3 | 测试补充 | 增加单元测试、集成测试 | ★★★★☆ |
| 4 | 性能优化 | 算法优化、内存优化 | ★★★☆☆ |
| 5 | 新功能开发 | 实现新特性 | ★★★☆☆ |
| 6 | 代码重构 | 改善代码结构 | ★★☆☆☆ |

## Issue筛选策略

### 优先选择的Issue特征
- 标签含 `good first issue` / `help wanted` / `documentation`
- 已有明确复现步骤的bug
- 维护者已确认需要修复的issue
- 范围小、边界清晰的改进

### 避免选择的Issue特征
- 需要物理实验验证的
- 涉及核心物理模型修改的（除非有明确spec）
- 长期未响应的stale issue
- 需要大量领域专家讨论的设计决策

## PR提交节奏

1. **第一个PR**：选择最简单的贡献（文档/bug fix），建立信任
2. **后续PR**：逐步增加复杂度
3. **每个PR间隔**：等待上一个PR获得反馈后再提交下一个
4. **同时进行**：最多2个open PR

## 核工程领域贡献要点

作为核工程+代码专家贡献时：

1. **物理正确性**：任何涉及物理计算的修改必须验证正确性
2. **单位一致性**：核工程代码对单位极其敏感，必须确保SI单位制
3. **安全裕度**：涉及安全参数时，保守优于激进
4. **可追溯性**：代码注释引用相关文献或标准
5. **数值精度**：浮点运算注意精度损失，关键路径使用高精度
