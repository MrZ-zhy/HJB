# PR策略

## 贡献类型优先级

> 按AI可完成度和贡献价值排序

| 优先级 | 贡献类型 | 说明 | AI适合度 |
|--------|---------|------|---------|
| 1 | Bug修复 | 修复已知bug，范围明确 | ★★★★★ |
| 2 | 文档完善 | 补充API文档、教程、示例 | ★★★★★ |
| 3 | 测试补充 | 增加单元测试、集成测试 | ★★★★☆ |
| 4 | 性能优化 | 算法优化、内存优化 | ★★★☆☆ |
| 5 | 新功能开发 | 实现新特性 | ★★★☆☆ |
| 6 | 代码重构 | 改善代码结构 | ★★☆☆☆ |

## 自适应优先级

> 参考强化学习优先经验回放（PER）：不按固定优先级处理，而是基于历史贡献结果动态调整。

### 优先级调整因子

| 因子 | 影响 | 说明 |
|------|------|------|
| 历史合并率 | 正向 | 该仓库/类型的历史PR合并率越高，优先级越高 |
| 维护者响应速度 | 正向 | 平均review响应时间越短，优先级越高 |
| 预测合并概率 | 峰值 | 预测合并率在40%-60%的Issue最有学习价值，优先处理 |
| 等待时间 | 正向 | Issue在队列中等待超过7天，强制提升优先级（防饥饿） |
| CI通过率 | 正向 | 该仓库CI首次通过率越高，优先级越高 |

### 优先级计算公式

```
动态优先级 = 基础优先级 × 历史合并率权重 × 响应速度权重 × 学习价值权重 + 等待时间补偿
```

### 策略更新周期

- 每10个PR完成后，重新评估优先级权重
- 每个贡献周期（周）结束时，输出策略效果报告
- 当PR合并率连续3次低于SLO时，自动切换到保守策略

### 保守策略 vs 进取策略

| 维度 | 保守策略 | 进取策略 |
|------|---------|---------|
| 触发条件 | 合并率<30%或连续被拒 | 合并率>60%且CI通过率>80% |
| Issue选择 | 仅选good-first-issue和文档类 | 可选中等复杂度Issue |
| PR节奏 | 等待上一个PR合并后再提交下一个 | 最多2个并行open PR |
| 仓库选择 | 优先贡献门槛最低的仓库 | 可尝试门槛较高的仓库 |
| 代码范围 | 最小变更集 | 可提交较大功能改进 |

## Issue筛选策略

### 优先选择的Issue特征
- 标签含 `good first issue` / `help wanted` / `documentation`
- 已有明确复现步骤的bug
- 维护者已确认需要修复的issue
- 范围小、边界清晰的改进
- 近期有维护者评论的issue（表明项目活跃）

### 避免选择的Issue特征
- 需要物理实验验证的
- 涉及核心物理模型修改的（除非有明确spec）
- 长期未响应的stale issue（>90天无活动）
- 需要大量领域专家讨论的设计决策
- 已有其他贡献者认领的issue

### 分支保护感知

> 提交PR前自动检测目标仓库的分支保护规则，据此调整策略。

| 保护规则 | 检测方式 | 策略调整 |
|---------|---------|---------|
| 需要N个审批 | 读取仓库settings | 标注"需要社区审查"，预期合并周期更长 |
| 需要CI通过 | 读取required checks | 确保本地CI通过后再提交 |
| 需要CODEOWNERS审批 | 读取CODEOWNERS文件 | 预期审批周期更长，降低优先级 |
| 禁止force-push | 读取分支保护 | 修复CI时使用新commit而非force-push |
| 要求线性历史 | 读取merge策略 | 使用rebase而非merge commit |

## WIP限制规则

> 参考Kanban WIP限制：防止贡献系统过载。

| 维度 | 限制 | 超限处理 |
|------|------|---------|
| 全局活跃PR数 | ≤ 5 | 暂停新PR提交，等待最早PR获得反馈 |
| 单仓库活跃PR数 | ≤ 1 | 跳过该仓库，选择下一个优先级项目 |
| 待响应Review数 | ≤ 3 | 暂停新PR提交，优先处理Review反馈 |
| 每周新PR提交数 | ≤ 3 | 控制节奏，避免维护者感知为"刷屏" |

### 金丝雀式贡献

> 参考金丝雀部署：先向低关注度仓库试提交，验证流程通过后再向高影响力仓库扩展。

```
第1-2个PR → 低影响力仓库（如fusion-sim、OpenReactor）
    │
    ├─ 合并率≥50% → 逐步向中影响力仓库扩展（PlasmaGym、FUSE）
    │
    └─ 合并率<50% → 停留在低影响力仓库，分析原因并优化策略
         │
         ├─ 优化后合并率≥50% → 继续扩展
         │
         └─ 持续<50% → 切换到保守策略
```

## PR提交节奏

1. **第一个PR**：选择最简单的贡献（文档/bug fix），建立信任
2. **后续PR**：逐步增加复杂度
3. **每个PR间隔**：等待上一个PR获得反馈后再提交下一个
4. **同时进行**：最多2个open PR（进取策略下）
5. **节奏感知**：如维护者响应慢（>7天），降低提交频率

## Dependabot式PR模板

> 参考Dependabot的PR结构：清晰、结构化、信息完整。

### PR标题格式

```
<type>(<scope>): <description>

示例：
fix(solver): resolve token expiration edge case in transport simulation
docs(api): add missing API documentation for config module
test(physics): add unit tests for Bosch-Hale fusion reaction rate
```

- [ ] 遵循Conventional Commits

## 错误预算机制

> 参考SRE错误预算：定义贡献质量SLO，预算耗尽时暂停提交并转入可靠性模式。

### 贡献SLO定义

| SLO指标 | 目标值 | 计算窗口 | 说明 |
|---------|--------|---------|------|
| PR合并率 | ≥ 30% | 滚动30天 | 被合并的PR占总提交PR的比例 |
| CI首次通过率 | ≥ 80% | 滚动30天 | PR提交后CI首次运行通过的比例 |
| Review响应时间 | ≤ 48h | 滚动30天 | 维护者首次Review的平均响应时间 |
| 代码退回率 | ≤ 10% | 滚动30天 | Review后要求重大修改的PR比例 |

### 错误预算计算

```
错误预算 = (1 - SLO目标) × 窗口内PR数

例：合并率SLO=30%，30天内提交10个PR
错误预算 = (1-0.3) × 10 = 7 个PR可被拒绝

当被拒绝PR数 ≥ 7 时，错误预算耗尽
```

### 预算耗尽处理

```
IF 错误预算耗尽:
    1. 暂停所有新PR提交
    2. 进入可靠性模式：
       - 分析所有被拒绝PR的原因
       - 识别系统性问题（代码质量？Issue选择？仓库选择？）
       - 制定改进计划
    3. 改进计划执行完毕后，重置错误预算
    4. 以保守策略重新开始贡献
```

## 核工程领域贡献要点

作为核工程+代码专家贡献时：

1. **物理正确性**：任何涉及物理计算的修改必须验证正确性
2. **单位一致性**：核工程代码对单位极其敏感，必须确保SI单位制
3. **安全裕度**：涉及安全参数时，保守优于激进
4. **可追溯性**：代码注释引用相关文献或标准
5. **数值精度**：浮点运算注意精度损失，关键路径使用高精度
6. **边界条件**：等离子体物理代码对边界条件极其敏感，必须仔细验证
7. **收敛性**：迭代求解器必须验证收敛性，避免虚假收敛

## 各项目特定策略

> 在具体项目评估文件中补充

### TORAX（Google DeepMind）
- 语言：Python (JAX)
- 重点：文档、测试、物理模型改进
- 特殊注意：Google内部审核流程可能较长，预期合并周期7-14天
- 分支保护：预计需要1个审批+CI通过

### OpenReactor
- 语言：Go
- 重点：控制逻辑、监控功能、文档
- 特殊注意：个人项目，PR响应快，适合金丝雀式贡献的起点
- 分支保护：预计较宽松

### FUSE
- 语言：Julia
- 重点：物理模型、工程计算模块
- 特殊注意：Julia生态特殊，注意版本兼容性
- 分支保护：企业项目，可能需要正式审批流程

### OpenMC
- 语言：Python/C++
- 重点：中子学模拟、后处理工具
- 特殊注意：社区活跃，CI要求严格，确保本地测试通过后再提交
- 分支保护：预计需要CI通过+1个审批

### PlasmaGym
- 语言：Python
- 重点：RL环境、奖励函数、基线算法
- 特殊注意：AI+聚变交叉领域，AI代码生成优势最大
- 分支保护：社区项目，预计较宽松

### OpenFUSIONToolkit
- 语言：Python/Fortran
- 重点：Python接口改进、文档、测试
- 特殊注意：修改Fortran核心需谨慎，优先在Python层贡献
- 分支保护：学术项目，预计需要维护者审批

### ITER-IMAS
- 语言：Fortran/Python/C++
- 重点：Python封装、文档翻译、安装脚本
- 特殊注意：国际组织项目，贡献流程正式，审核严格
- 分支保护：预计需要多轮审批

### fusion-sim
- 语言：Rust/WASM/React
- 重点：React前端、文档、新装置配置
- 特殊注意：Rust代码AI生成质量较低，优先在前端和文档方向贡献
- 分支保护：个人项目，预计较宽松
