# 开发链条模板

> 此模板定义了每个项目的标准开发链条结构。
> 实际开发时根据项目特点调整，但必须遵循此框架。

## 标准开发链条

```
[项目根]
 │
 ├──⏳ P1-环境搭建 ── fork仓库、clone源码、配置开发环境
 │   │
 │   ├──⏳ P1.1-代码研读 ── 阅读项目结构、核心模块、文档
 │   │   │
 │   │   ├──⏳ P1.1.1-架构理解 ── 绘制模块依赖关系图
 │   │   │   │
 │   │   │   └──⏳ P1.1.2-Issue筛选 ── 找到适合AI贡献的Issue
 │   │   │
 │   │   └──⏳ P1.2-本地验证 ── 运行测试套件，确认环境可用
 │   │
 │   ├──⏳ P2-开发实施 ── 根据选定Issue进行开发
 │   │   │
 │   │   ├──⏳ P2.1-方案设计 ── 设计实现方案，记录在链条中
 │   │   │   │
 │   │   │   ├──⏳ P2.1.1-主方案 ── 首选实现路径
 │   │   │   │   └──🔀 P2.1.2-备选方案 ── 如主方案受阻的替代路径
 │   │   │   │
 │   │   │   └──⏳ P2.2-编码实现 ── 编写代码
 │   │   │
 │   │   ├──⏳ P2.3-测试验证 ── 运行测试、手动验证
 │   │   │
 │   │   └──⏳ P2.4-代码审查 ── 自检代码质量
 │   │
 │   ├──⏳ P3-PR提交 ── 创建Pull Request
 │   │   │
 │   │   ├──⏳ P3.1-分支整理 ── 整理commit、rebase
 │   │   │   │
 │   │   │   └──⏳ P3.2-PR创建 ── 推送分支、创建PR
 │   │   │
 │   │   └──⏳ P3.3-进度更新 ── 更新进度表，标记新一轮开发状态为「已完成」
 │   │
 │   └──⏳ P4-后续跟进 ── 跟踪PR审核状态
 │       │
 │       ├──⏳ P4.1-响应反馈 ── 根据review意见修改
 │       │
 │       └──⏳ P4.2-合并确认 ── 确认PR被合并
```

## 节点信息模板

每个节点在激活时，需记录以下信息：

```yaml
节点ID: P2.1.1
节点名称: 主方案设计
状态: 🔄进行中
开始时间: YYYY-MM-DD HH:MM
完成时间: —
父节点: P2.1
子节点: P2.2
产出物: 方案设计文档/代码文件路径
备注: 具体实现细节
```

## 链条生长规则

1. **顺序推进**：默认按P1→P2→P3→P4顺序推进
2. **条件分支**：当主方案受阻时，创建🔀分支节点
3. **分支合并**：分支成功后合并回主线，失败则标记❌
4. **回溯**：允许回退到上一节点重新执行
5. **并行**：同一层级的不同分支可并行（但AI一次只执行一条路径）

## 链条与进度表的映射

- 链条当前活跃节点 → 进度表的 `CURRENT_CHAIN_NODE`
- 链条整体阶段 → 进度表的 `DEVELOP_PHASE`
- 链条完成状态 → 进度表的 `新一轮开发状态`
