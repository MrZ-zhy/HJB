# V4 架构 · 第一性原理

> **目的**：用第一性原理推导 V4 设计，每一层决策都可追溯到"为什么"。
> 取代 v1（脚本集合）/ v2（5 步 prompt + 4 脚本）/ v2.1（多项目并进机制 + 决策矩阵）三大版本。

---

## 1. 第一性原理：目标 → 约束 → 设计

### 1.1 终极目标（G）

> **持续、可靠、低人工介入地**向核聚变开源项目贡献代码。

### 1.2 不可妥协的约束（C）

| # | 约束 | 推导 |
|---|------|------|
| C1 | 每次 tick 是新隔离 shell（无持久内存） | 沙盒环境：环境变量/进程状态不跨调用 |
| C2 | 唯一持久化介质 = GitHub（git push 后即落） | 沙盒本地文件系统是临时的 |
| C3 | 单 AI agent，单 tick 时间窗有限 | prompt 长度 + token 预算有限 |
| C4 | 多个项目必须并进 | 单仓 PR ≤ 1 + 干等 review 会死锁 |
| C5 | 失败必须可观察 | 沙盒无 stdout 重定向，错误只能从进度表/日志读 |
| C6 | 系统必须自描述 | 下次 tick 的 AI 不能依赖前次上下文 |

### 1.3 设计公理（A）

> **A1**：一次 tick = 一次原子 commit。即"做了什么"与"落盘"必须原子绑定（C2）。
> **A2**：状态机是代码，不是文档。文档是 spec，代码是 implementation（C6）。
> **A3**：失败可隔离。一个 strategy 失败不能污染其他（C5）。
> **A4**：决策可审计。每次决策的"为何"必须能追溯到数据（C5）。
> **A5**：可插拔 = 可演进。决策规则不能写死，否则每次新约束都要改核心（C1+C4）。

### 1.4 V4 → 设计映射

| 公理 | V4 实现 |
|------|--------|
| A1 一次 tick 一次原子 commit | `core.orchestrator.tick()` 串行执行 7 步，**任一步失败回滚**到 LAST_HEARTBEAT |
| A2 状态机是代码 | `core.state_machine.StateMachine`：14 状态 + 转换表 + 守卫函数 |
| A3 失败可隔离 | `strategies.base.Strategy` 协议 + `try/except` 包每个 strategy |
| A4 决策可审计 | `observability.structured_log` 记录 `decision_taken → reason → data` |
| A5 可插拔 | `strategies/` 目录新加 strategy 文件即注册（reflection） |

## 2. V4 工作流：7 步

```
[Trae Tick 触发]  --cron-->  [Orchestrator.tick()]
                                     │
   ┌─────────────────────────────────┴─────────────────────────────┐
   │                                                              │
   ▼                                                              ▼
 Step 1: ENV_PREPARE                                       失败 → emit preflight_failed
   - git pull --rebase                                              ↓
   - 注入 GITHUB_TOKEN                                       Step 4: STATE_DECIDE
   - 探测 git 身份                                                  │
                                                                ▼
 Step 2: PREFLIGHT                                     Step 5: EXECUTE
   - token scopes                                              选 strategy → 调 action
   - upstream 可达                                                    │
   - fork 存在                                                   ▼
   - 本地工作区干净                                          Step 6: PERSIST
   - 进度表健康                                                  原子写主表 + 子表 + commit + push
                                                                │
                                                                ▼
 Step 3: LOAD_STATE                                  Step 7: REPORT
   - parse 主表 → typed state                              结构化 JSON 输出
   - parse 所有子表 → per-project state
```

每步输出 `StepResult(step, ok, elapsed_ms, payload)`，失败不中断（除非 Step 6 persist 失败）。

## 3. V4 模块划分

```
核聚变开源贡献系统/自动化PR系统核心/scripts/v4/
├── engine.py              # 统一 CLI 入口（命令: tick / status / report / dry-run）
├── core/
│   ├── __init__.py
│   ├── orchestrator.py   # 7 步工作流的状态机驱动
│   ├── state_machine.py  # 14 状态贡献状态机实现（V2.1 文档→代码）
│   ├── event_bus.py      # 解耦 strategy 与 persistence 的事件总线
│   └── models.py         # 数据类：EngineState, ProjectState, Action, StepResult
├── strategies/
│   ├── __init__.py
│   ├── base.py           # Strategy Protocol + 注册机制
│   ├── decision_matrix.py# 多项目并进决策（v2.1 决策矩阵规则化）
│   ├── pr_strategy.py    # PR 触发器（v2 strategy_evaluator 升级）
│   ├── project_selector.py# 项目选择（v2 project_rotator 升级）
│   └── health_check.py   # preflight（v2 preflight 升级）
├── persistence/
│   ├── __init__.py
│   ├── progress_table.py # 主表读写（继承 v2.1 engine_helper 的 parse + 治本 regex）
│   ├── project_progress.py# 子表读写（继承 v2.1 engine_helper 的 project_parse/get/set）
│   └── git_ops.py        # git 原子操作（commit + push + rebase）
├── observability/
│   ├── __init__.py
│   ├── structured_log.py # JSON Lines 日志（stdout + 进度表双写）
│   └── metrics.py        # RED 指标 + SLO 燃烧率
└── tests/
    ├── __init__.py
    ├── test_state_machine.py
    ├── test_decision_matrix.py
    └── test_progress_table.py
```

### 3.1 旧脚本如何映射

| 旧 v2 脚本 | V4 模块 | 变化 |
|------------|---------|------|
| `engine_helper.py` | `persistence/progress_table.py` + `persistence/project_progress.py` | 同名 API 内嵌；不再有顶层 CLI（V4 统一走 engine.py） |
| `preflight.py` | `strategies/health_check.py` | 升级为 Strategy 实现，输出结构化 StepResult |
| `strategy_evaluator.py` | `strategies/pr_strategy.py` | 升级为 Strategy 实现，可热替换规则 |
| `project_rotator.py` | `strategies/project_selector.py` | 升级为 Strategy 实现，决策 = 走决策矩阵 |

### 3.2 铁律映射（v1 1-15 + v2 16-20 → V4 1-25）

V4 把铁律分三组：
- **系统铁律 (S1-S8)**：1/2/13/14/15/16/19/20（v2 保留）
- **状态机铁律 (M1-M7)**：v2 状态机文档的合法转换表 → 代码
- **策略铁律 (P1-P10)**：v2.1 决策矩阵 + WIP/预算 → 代码
- 总数 25 条，V4 progress table codeblock 单字段 `IRON_LAWS` 集中存放

## 5. V4 与 v2 关键差异

| 维度 | v2.1 | V4 |
|------|------|----|
| 入口 | 4 个独立脚本 | `engine.py tick` 单一入口 |
| 状态机 | 文档 | 代码（`core.state_machine`） |
| 决策矩阵 | 文档 + 硬编码 if-else | 规则注册表（`strategies/decision_matrix`） |
| 错误隔离 | 失败 = 整个 tick 死 | per-strategy try/except |
| 可观察性 | 仅 LAST_HEARTBEAT 字符串 | JSON Lines + metrics snapshot |
| 可插拔 | 无 | Strategy protocol + 自动注册 |
| 可审计 | 无 | 每次决策带 `reason` + `data` |
| 心跳 | 字符串 | `Heartbeat(version=4, sha=..., step=..., ...)` |
| 项目状态 | 主表"当前开发项目" | 每项目子表（v2.1 已落地） |
| 多项目 | 决策矩阵 | 同上，V4 把它转成 rules |

## 6. V4 退化路径

V4 设计为可向 v2 退化（不破坏数据兼容）：

| 退化场景 | V4 行为 |
|---------|--------|
| V4 引擎不可用 | 进度表 LOCK=true，回退到 v2 prompt（手动） |
| 子表丢失 | 重写子表（V4 自带 `project-init` CLI） |
| 主表 codeblock 解析失败 | 走 v2 兼容的 `_KV_COMMENT_MARKER_RE` 治本 split |
| GitHub API 失败 | 重试 3 次后写 `LAST_HEARTBEAT_STATUS=api_degraded`，不阻塞下次 |

## 7. 何时升 V5

V4 完成以下后才考虑 V5：
- ✅ 至少 5 个 PR 走完 V4 全生命周期（创建 → review → 合并/拒绝）
- ✅ decision_matrix 规则命中 ≥ 1 次真实触发
- ✅ 至少 2 个项目通过 V4 完成 on-board
- ✅ Trae 调度连续 7 天无 preflight_failed
- ✅ 用户验收 V4 报告格式

**否则不升 V5。** V4 的核心是"稳定可演进"，不是"新潮"。

## 8. 第一性原理检查清单（每次新设计前必过）

- [ ] 这个设计服务于 G（终极目标）吗？
- [ ] 它违反了哪条 C？如有，能否改设计而非绕开？
- [ ] 它满足哪条 A？如都不满足，删除之。
- [ ] 它能在 C1（无状态 shell）下工作吗？
- [ ] 它能在 C5（失败可观察）下被诊断吗？
- [ ] 它能在 C6（自描述）下被下次 tick 接手吗？

**任何"是"的答案设计都不通过此检查。**
