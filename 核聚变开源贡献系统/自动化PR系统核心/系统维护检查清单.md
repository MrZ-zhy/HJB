# 系统维护检查清单

> 当新一轮开发状态为「进行中」时，定时任务执行此清单

## 检查流程

### 第一步：进度表完整性验证

- [ ] 进度表.md文件存在且可读
- [ ] 新一轮开发状态字段存在且值为「进行中」或「已完成」
- [ ] 当前开发项目信息完整（项目名称、仓库、分支）
- [ ] 链状开发进度树结构完整
- [ ] 操作指令区字段完整（NEXT_ACTION, CURRENT_CHAIN_NODE, DEVELOP_PHASE, LOCK）

### 第二步：项目源码同步

- [ ] 项目目录存在
- [ ] 执行 `git fetch origin` 获取远程更新
- [ ] 检查是否有新commit：`git log HEAD..origin/main --oneline`
- [ ] 如有新commit，执行 `git pull --rebase`
- [ ] 检查是否有冲突，如有则记录在链条中

### 第三步：分支状态检查

- [ ] 当前分支名与进度表记录一致
- [ ] 分支未落后远程分支
- [ ] 工作区干净（无未提交修改）

### 第四步：PR状态跟踪

- [ ] 检查已提交PR的状态
- [ ] 如PR已被review，记录反馈意见
- [ ] 如PR需要修改，将修改任务加入链条
- [ ] 如PR已合并，记录合并信息

### 第五步：更新维护状态

- [ ] 更新进度表中的系统维护状态表
- [ ] 更新 LAST_UPDATE 日期
- [ ] 如发现异常，在操作指令区标记

## 异常处理

| 异常情况 | 处理方式 |
|---------|---------|
| 进度表损坏 | 从备份恢复或重建 |
| 源码同步冲突 | 记录冲突，等待开发轮次处理 |
| PR被关闭 | 分析原因，决定是否重新提交 |
| 分支丢失 | 从远程重新拉取 |
| 定时任务失败 | 检查LOCK状态，必要时重置 |

## 维护完成标志

所有检查项完成后，在进度表系统维护状态中更新时间戳。
不修改新一轮开发状态（保持「进行中」）。
